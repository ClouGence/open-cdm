# 代码风格和格式化风格

## 文档目的

本文档用于约束 AI 在本仓库中的代码风格判断边界。
重点是规定什么必须遵守、什么不允许做；不规定具体编码技巧。

## 事实来源

- Java 格式化风格以仓库根目录的 codeformat.xml 为准。
- 当前格式化配置体现的是 4 空格缩进、保留现有换行结构、使用 @formatter:off / @formatter:on 控制局部关闭格式化。
- 仓库整体风格明显偏向“遵循现有代码形态”，而不是强制统一成全新样式。

## 必须遵守的边界

- 必须优先服从现有文件的局部风格、命名和排版，而不是引入新的个人偏好。
- 必须最小化格式化影响，避免无关行的大面积重排。
- 必须尊重已有 formatter on/off 边界，不跨越这些边界重排代码。
- 必须保持模块内风格一致，不在单个改动中混入多种排版风格。

## 不允许的行为

- 不允许借机做纯风格性重构，例如批量改 import 排序、括号风格、空行数量、注释形态。
- 不允许为了“更现代”或“更整洁”而重写现有命名、字段可见性、类布局。
- 不允许把代码风格文档升级成静态分析规则清单、review 清单或强制重写计划。
- 不允许以代码风格为名跨模块批量改动无关文件。

## AI 需要特别注意的事项

- 若已有标准 getter/setter、Lombok、已有字段访问方式或现有 API 风格，应优先保持一致。
- 若现有类使用 public field、手写 getter/setter、builder、注解驱动等特定模式，AI 不应默认替换成另一种模式。
- 若用户只要求修功能或修边界，不应顺带调整风格。
- 与风格相关的判断，只能在现有仓库事实基础上做总结，不得凭空制定新的“团队强制规范”。
